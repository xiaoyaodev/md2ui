# md2ui 链接处理方案

## URL 结构

站内文档 URL 采用 pathname 放 hash、原生 hash 放锚点的格式：

```
http://localhost:3000/kX9mN2pQ              （文档）
http://localhost:3000/kX9mN2pQ#快速开始      （文档 + 锚点）
```

- 路径部分：文档 key 的 FNV-1a 双轮 hash，base62 编码，取前 8 位
- 锚点部分：github-slugger 生成的语义化 slug（中文保留原文）

## Hash 算法

- 算法：FNV-1a 双轮（两个不同 seed），合并为 64 bit
- 编码：base62（0-9a-zA-Z），8 字符
- 碰撞空间：62^8 ≈ 218 万亿
- 依赖：零（纯 JS 实现，替代了 crypto-js/md5）

## 锚点 ID 生成

使用 [github-slugger](https://www.npmjs.com/package/github-slugger) 生成标题锚点：

```
## Hello World        →  id="hello-world"
## 快速开始            →  id="快速开始"
## API 参考            →  id="api-参考"
## FAQ                →  id="faq"
## FAQ (重复)          →  id="faq-1"
```

优势：与 GitHub 行为一致，语义清晰，重复标题自动去重。

注意：锚点链接（`#xxx`）和 `.md#anchor` 中的 anchor 部分，在 renderer.link 中直接使用原始值，不经过 slugger 二次转换，避免计数器追加后缀导致不匹配。

## 链接分类与处理

### 一、站外链接（不拦截，新标签页打开）

| 小类 | 示例 | 处理方式 |
|------|------|----------|
| http/https | `[GitHub](https://github.com)` | `target="_blank"` 新标签页 |
| mailto | `[邮件](mailto:x@x.com)` | 浏览器默认行为 |
| tel | `[电话](tel:123)` | 浏览器默认行为 |
| 非 .md 相对链接 | `[图片](./a.png)` | `target="_blank"` 新标签页 |

### 二、站内链接（拦截，SPA 内导航）

| 小类 | markdown 写法 | 渲染后 HTML | 处理方式 |
|------|--------------|-------------|----------|
| 纯锚点 | `[xxx](#标题)` | `data-anchor="标题"` | 拦截，scrollToHeading |
| 同目录 .md | `[xxx](./文档.md)` | `data-doc-key="key"` | 拦截，loadDoc |
| 跨目录 .md | `[xxx](../目录/文档.md)` | `data-doc-key="key"` | 拦截，loadDoc |
| .md 带锚点 | `[xxx](./文档.md#标题)` | `data-doc-key` + `data-anchor` | loadDoc + scroll |
| 目标不存在 | `[xxx](./不存在.md)` | `class="broken-link"` | 不跳转，标记失效 |

## 实现方式：渲染阶段改写

在 `marked` 的自定义 renderer 中统一处理：

- `renderer.heading`：使用 github-slugger 生成语义化 ID
- `renderer.link`：分类处理站内/站外链接，生成 data 属性
- 点击拦截只读 `data-*` 属性，不做路径解析

### 处理流程

```
markdown 原文
    |
    v
marked renderer.link(href, title, text)
    |
    +-- 站外？(http/mailto/tel) --> <a target="_blank">
    |
    +-- 纯锚点？(#xxx) --> <a data-anchor="slug">
    |
    +-- .md 链接？ --> 解析相对路径 --> 查找文档 --> hash(key)
    |       |
    |       +-- 找到 --> <a href="/hash#anchor" data-doc-key="key">
    |       +-- 找不到 --> <a class="broken-link">
    |
    +-- 其他相对链接 --> <a target="_blank">
```

## 路由机制

- URL 格式：`/{8位base62hash}#{slug锚点}`
- 前进/后退：监听 `popstate` 事件，通过 `history.state.scrollTop` 恢复滚动位置
- SPA fallback：服务端中间件将非静态资源请求重写到 index.html
- 页面刷新：从 `window.location.pathname` 读取 hash，查找对应文档

### history 状态管理

所有 `pushState` / `replaceState` 均携带 `{ scrollTop }` 状态，用于浏览器回退时精确恢复滚动位置。

| 场景 | history 操作 | scrollTop |
|------|-------------|-----------|
| 用户滚动页面 | replaceState（同步锚点到 URL） | 实时记录当前位置 |
| 点击文档内锚点链接 | replaceState 旧条目 → pushState 新条目 | 旧条目保存跳转前位置 |
| 点击 TOC 目录项 | replaceState | 不产生历史条目（导航辅助） |
| 点击侧边栏切换文档 | replaceState 旧条目 → pushState 新条目 | 旧条目保存当前位置 |
| 点击跨文档锚点链接 | pushState 文档 → replaceState 追加锚点 | 新文档从顶部开始 |
| 用户点击 Home | replaceState 旧条目 → pushState / | 旧条目保存当前位置 |
| 浏览器回退（同文档锚点） | 不操作 history | 从 state.scrollTop 恢复 |
| 浏览器回退（跨文档） | replaceState（keepState 保留 state） | 从 state.scrollTop 恢复 |
| 浏览器回退（到首页） | 不操作 history | 显示欢迎页 |
| 首次加载 / F5 刷新 | replaceState | 滚动到锚点或顶部 |

### 锚点链接处理

纯锚点链接（`#xxx`）和 `.md#anchor` 中的 anchor 部分，在 renderer.link 中直接使用原始值，不经过 `slugger.slug()` 二次转换。原因：slugger 实例内部维护计数器，标题渲染时已注册过该 slug，锚点链接再次调用会被追加 `-1` 后缀导致不匹配。
